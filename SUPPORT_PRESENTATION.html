⚙️ Slide 6 : Fonctionnalités Clés

🎯 À dire :

"L'application offre plusieurs fonctionnalités importantes pour faciliter l'accès aux données :"

1. Extraction automatique des métadonnées

"Au démarrage, l'application lit tous les fichiers XML présents dans le dossier 'data' et extrait automatiquement :

  • Les informations générales (titre, ID, abstract)
  • La période et la couverture géographique
  • Les producteurs, financeurs et auteurs
  • La méthodologie et le plan d'échantillonnage
  • Les conditions d'accès et le copyright
  • Le nombre de variables et d'observations

2. Recherche en temps réel

"Une barre de recherche intelligente permet de trouver rapidement les enquêtes. La recherche s'effectue dans les titres, résumés et mots-clés, avec des résultats qui s'affichent instantanément pendant que vous tapez."

3. Pages détaillées avec onglets

"Chaque enquête dispose d'une page complète organisée en 5 onglets :

  • Vue d'ensemble : Description, couverture, mots-clés
  • Méthodologie : Plan d'échantillonnage, mode de collecte
  • Production : Producteurs, financeurs, contacts
  • Ressources externes : Fichiers de données, questionnaires, rapports
  • Accès aux données : Conditions d'utilisation, demande d'accès

4. Ressources externes

"Une innovation importante : la possibilité d'ajouter des liens vers des ressources complémentaires via un simple fichier JSON. Pour chaque enquête, on peut proposer :

  • Les fichiers de microdonnées (SPSS, Stata, CSV)
  • Les questionnaires et manuels d'enquêteurs
  • Les rapports et publications
  • La documentation technique (codebooks, guides)

⚙️ Slide 9 : Aspects Techniques

🎯 À dire :

"Techniquement, l'application repose sur une architecture simple mais efficace :"

1. Extraction des métadonnées (Backend)

"Au cœur du système, une classe Python appelée DatasetParser utilise la bibliothèque lxml pour parser les fichiers XML DDI. Elle extrait automatiquement toutes les métadonnées en utilisant des requêtes XPath sur la structure XML."

# Exemple simplifié du parser class DatasetParser: def parse_metadata(self): # Extraction du titre title = self.get_text('//ddi:stdyDscr/ddi:citation/ddi:titlStmt/ddi:titl') # Extraction des dates start_date = self.get_text('//ddi:collDate[@event="start"]/@date') # Extraction des producteurs producers = self.get_all_text('//ddi:producer') # ... et ainsi de suite pour toutes les métadonnées

2. Système de routage Flask

"Flask gère le routage des URLs. Chaque page correspond à une route Python :

  • / → Page d'accueil
  • /catalog → Liste des datasets
  • /dataset/<id> → Page détail
  • /search → API de recherche (JSON)

3. Génération dynamique avec Jinja2

"Les templates Jinja2 permettent de générer le HTML dynamiquement. Par exemple, pour afficher tous les datasets, on fait simplement une boucle sur le dictionnaire des données chargées."

4. Configuration JSON pour les ressources

"Les ressources externes sont gérées via un fichier JSON simple. Pour chaque dataset, on liste les fichiers disponibles avec leurs URLs, descriptions et métadonnées. C'est flexible et facile à maintenir."

📊 Slide 10 : Flux de Données

🎯 À dire :

"Voyons maintenant comment les données circulent dans l'application, du fichier XML jusqu'à l'affichage dans le navigateur :"

Cycle de vie complet

1. Démarrage de l'application

Python lance Flask et charge tous les fichiers XML du dossier 'data'

2. Parsing des XML

Chaque fichier est parsé et les métadonnées sont extraites et stockées en mémoire

3. Chargement des ressources

Le fichier resources.json est lu et les liens sont associés aux datasets

4. Serveur prêt

Flask écoute sur le port 5001, prêt à recevoir des requêtes

5. Requête utilisateur

L'utilisateur visite une page → Flask génère le HTML → Envoi au navigateur

⚡ Performance

Les données ne sont chargées qu'une seule fois au démarrage. Ensuite, tout est en mémoire, ce qui permet des temps de réponse très rapides. Pour modifier les données, il suffit de redémarrer l'application.

❓ Slide 15 : Questions & Réponses - Préparation

💡 Questions probables et réponses préparées

Question 1 : "Pourquoi Flask et pas Django ?"

Réponse :

"J'ai choisi Flask pour sa simplicité et sa légèreté. Pour ce projet, nous n'avons pas besoin des fonctionnalités lourdes de Django comme l'ORM ou le système d'admin. Flask permet de créer rapidement une application web fonctionnelle sans surcharge. De plus, c'est idéal pour apprendre les bases du développement web en Python."

Question 2 : "Comment gérez-vous les mises à jour des données ?"

Réponse :

"Actuellement, pour ajouter ou modifier des données, il suffit de placer les nouveaux fichiers XML dans le dossier 'data' et redémarrer l'application. Elle recharge automatiquement tout. Pour une version production, on pourrait implémenter un système de rechargement à chaud ou un dashboard admin pour gérer les datasets sans redémarrage."

Question 3 : "Est-ce que ça peut gérer beaucoup de datasets ?"

Réponse :

"Dans la version actuelle, les données sont chargées en mémoire, donc c'est limité par la RAM disponible. Pour quelques dizaines d'enquêtes, c'est largement suffisant. Pour des centaines ou milliers de datasets, il faudrait migrer vers une vraie base de données comme PostgreSQL, ce qui est une évolution naturelle du projet."

Question 4 : "Quelle est la différence avec NADA ?"

Réponse :

"NADA est un système complet développé par la Banque Mondiale avec de nombreuses fonctionnalités avancées. Mon projet est plus simple et ciblé : il se concentre sur la diffusion web de fichiers XML Nesstar existants. C'est une solution légère, facile à installer et à personnaliser, idéale pour des institutions qui ont déjà fait le travail avec Nesstar et veulent juste une meilleure interface web."

Question 5 : "Comment avez-vous appris Flask ?"

Réponse :

"J'ai utilisé la documentation officielle de Flask qui est excellente, combinée avec des tutoriels en ligne. Ce projet m'a permis d'apprendre en pratiquant : routage, templates Jinja2, parsing XML avec lxml. C'était une excellente opportunité d'appliquer concrètement nos connaissances Python dans un contexte de développement web."

Question 6 : "Temps de développement ?"

Réponse :

"Le développement s'est étalé sur environ 2-3 semaines : une semaine pour la structure de base et le parsing XML, une semaine pour l'interface et les fonctionnalités, et quelques jours pour les ressources externes, la documentation et les tests. Le plus long a été de bien comprendre la structure XML DDI et de créer un design professionnel."
